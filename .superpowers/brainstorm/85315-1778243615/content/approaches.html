Three Approaches — Client Pipeline System

Same goal, different architecture trade-offs. My recommendation is highlighted.

★ Recommended
Approach A
Expand the Admin Dashboard — New Tabs + Modal Wizards
All pipeline work happens inside the existing admin-dashboard.html. Add a "Leads" tab (the live Qualified Leads Register) with a "Prepare Quote" button per row that opens a 3-step modal wizard. After quote is sent, an "Onboard" button opens the 2-step onboarding wizard. The client-facing form is a separate public/client-onboarding.html page (already exists as a stub).
Pros
  • Zero new files — extends the existing pattern
  • All admin work in one place — no navigation between pages
  • Modal wizards keep context (lead data) without page reloads
  • Consistent with how CRM tab already works
  • Easiest to deploy — same standalone HTML file
Cons
  • admin-dashboard.html grows larger (already substantial)
  • Modal wizard JS is moderately complex
  • No URL-based navigation to a specific lead
New files: Edge Function supabase/functions/generate-pdf/index.ts · scripts/generate-reg02.cjs · updates to admin-dashboard.html + client-onboarding.html
Approach B
Separate Admin Pages per Stage
Split the admin into separate standalone HTML files: admin-leads.html (register + quote wizard), admin-onboarding.html (onboarding wizard + CRM). Each page is focused and smaller. A simple nav bar links between them.
Pros
  • Each file stays focused and manageable
  • URL-based navigation to each stage
  • Easier to extend individual pages independently
Cons
  • Auth state must be duplicated or shared across pages
  • More files to maintain and keep in sync
  • Shared components (nav, auth) need duplication
  • Breaks the single-file simplicity of the current admin
Approach C
Move Admin into the React SPA
Create React routes /admin/leads, /admin/quotes, /admin/onboarding etc. Full React component model, proper routing, shared state. The existing admin-dashboard.html is deprecated or migrated.
Pros
  • Most maintainable long-term
  • Full React ecosystem (hooks, context, routing)
  • Proper shared auth via AuthContext
Cons
  • Significant refactor — not what was asked for
  • Admin is publicly routed (needs careful auth guarding)
  • Much more scope than needed right now
  • YAGNI — existing standalone approach works well
